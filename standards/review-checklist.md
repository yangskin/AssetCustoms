# Code Review 检查单

在提交 PR 前与评审时可参考以下清单，确保质量一致性。

## 功能与正确性
- [ ] 需求与范围明确，边界条件覆盖。
- [ ] 无明显逻辑错误，失败路径可见且可恢复。
- [ ] 日志与错误信息有助于定位问题。
- [ ] 涉及 V1.1 的 FR/NFR 是否满足（见 `docs/requirements_v1.1.md`）。

## 可读性与维护性
- [ ] 代码简洁、命名准确、重复最小化。
- [ ] 有必要的 Docstring/注释解释“为什么”。
- [ ] 模块边界清晰，函数职责单一。

## 风格与规范
- [ ] 符合 Google Python Style 与项目编码规范（`standards/coding-style.md`）。
- [ ] 提交信息符合 Conventional Commits（`standards/commit-convention.md`）。

## 性能与资源
- [ ] 避免阻塞 UE 主线程的长耗时操作（必要时异步/分批）。
- [ ] 处理大批量资产时注意内存与时间复杂度。
- [ ] 典型资产“静默成功”流程 ≤ 5s（NFR1），若超出请说明原因与优化方向。

## 安全与健壮
- [ ] 外部输入/路径/环境变量做校验与防御。
- [ ] 不泄露敏感信息（密钥、机器信息等）。
- [ ] 失败时保留隔离区，不进行破坏性操作；遵循“静默成功，响亮失败”。

## 测试与验证
- [ ] 有最小可验证路径（手动/脚本）。
- [ ] 新增/变更逻辑有相应测试或验证说明（如适用）。

## 文档
- [ ] README/架构/路线图是否需要更新并已更新。
- [ ] 如有重要决策，是否新增/更新 ADR。
